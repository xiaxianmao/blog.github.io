滴滴面试总结

Posted on 2019-12-16 In 编码

今天去面试了滴滴，没有通过第二轮面试，内心十分遗憾。面试官最后对我说：“要自信点，不要总是说‘我觉得xxx’、‘我认为xxx’，这样会让人觉得你是在猜测，即使你可能知道也会觉得你了解不够深入。

还有就是大家对于“了解”或是“知道”的程度不一致，这样会导致我以为你很了解，其实你只是知道。这样的话可能考察的是你不擅长的领域，也就无法检测初面试者最真实的能力。

觉得很遗憾的是，我内心觉得这是一个很好的机会，一个独角兽的公司核心业务的开发的工作机会，没有把握住。不过仍旧非常感谢面试我的两位同仁，你们非常专业，也非常平易近人，是我对滴滴留下了比较好的印象。与我而言，我会快速补充相关知识，有机会我还要试一试。

现罗列一些我无法完全回答上来或回答的不好的问题，作为这次面试的总结。

第三题：时针分针在12点整相交，下一次相交在什么时刻

面试的没有头绪，因为回答的不够好，所以导致心理有些波动。回到家重新缕了下思路。

思路

一天时间86400s，半天为43200s。所以只需要判断每一秒的角度即可。四万多次直接遍历即可（可能不是最优解）

角度： 
    时针 1度 = 12h*3600s/360 = 120s
    分针 1度 = 1h*3600s/360 = 20s
    秒针 1度 = 60s/360 = 1/6s

算下每一秒各针对应的角度，时针和分针角度相同的时刻即为答案

调用

结论

0点之后下一次整数秒相交为12点，也就是不存在整数秒相交的情况。

判断非整数秒相交的情况

如果不存在整数秒相交，那么相交一定在上一秒到下一秒之间。则存在上一秒 时针的角度>分针的角度，下一秒时针的角度小于分针的角度，满足此条件即可达到相交的秒区间

调整代码后得出所有相交的时刻

方程式解法

设x秒时时针和分针角度相等，则有

x/120= (x - n * 3600)/10   43200>=x>=0,n取值范围为[0,11]

x = 12x - 12 * 3600n

11x = 12 * 3600n

x = 12 * 3600n / 11

得到的结果集

第四题 Ping

ping是个命令 用来检测网络是否能够到达某个站点或远端IP。

ping使用ICMP协议（Internet Control Message Protocol，控制报文协议），类型（8 ping 0 repley ping）。基于IP协议。

ICMP大致分成两种功能：差错通知和信息查询.

第八题：如何设计一个高可用的系统

高可用

冗余备份和失效转移解决高可用问题

服务层：负载均衡 注册发现 服务降级

应用层：无状态 负载均衡 分布式部署

数据层：冗余备份（冷，热备[同步，异步]，温备），失效转移。数据一致性 CPA

高并发

高性能的架构是以用户为中心，提供快速的网页访问体验，主要参数有较短的响应时间、较大的并发处理能力、较高的吞吐量与稳定的性能参数。

前端优化：网站业务逻辑之前的部分。

浏览器优化：减少 HTTP 请求数，使用浏览器缓存，启用压缩，CSS JS 位置，JS 异步，减少 Cookie 传输；CDN 加速，反向代理。

应用层优化：处理网站业务的服务器。使用缓存，异步，集群。

代码优化：合理的架构，多线程，资源复用（对象池，线程池等），良好的数据结构，JVM调优，单例，Cache 等。

存储优化：缓存、固态硬盘、光纤传输、优化读写、磁盘冗余、分布式存储（HDFS）、NoSQL 等。

可伸缩

应用层：对应用进行垂直或水平切分。然后针对单一功能进行负载均衡（DNS、HTTP[反向代理]、IP、链路层）。

服务层：与应用层类似。

数据层：分库、分表、NoSQL 等；常用算法 Hash，一致性 Hash。

可扩展

模块化，组件化：高内聚，低耦合，提高复用性，扩展性。

稳定接口：定义稳定的接口，在接口不变的情况下，内部结构可以“随意”变化。

设计模式：应用面向对象思想，原则，使用设计模式，进行代码层面的设计。

消息队列：模块化的系统，通过消息队列进行交互，使模块之间的依赖解耦。

分布式服务：公用模块服务化，提供其他系统使用，提高可重用性，扩展性。

安全

基础设施安全：硬件采购，操作系统，网络环境方面的安全。一般采用正规渠道购买高质量的产品，选择安全的操作系统，及时修补漏洞，安装杀毒软件防火墙。防范病毒，后门。设置防火墙策略，建立 DDOS 防御系统，使用攻击检测系统，进行子网隔离等手段。

应用系统安全：在程序开发时，对已知常用问题，使用正确的方式，在代码层面解决掉。防止跨站脚本攻击（XSS），注入攻击，跨站请求伪造（CSRF），错误信息，HTML 注释，文件上传，路径遍历等。还可以使用 Web 应用防火墙（比如：ModSecurity），进行安全漏洞扫描等措施，加强应用级别的安全。SQL注入

数据保密安全：存储安全（存储在可靠的设备，实时，定时备份），保存安全（重要的信息加密保存，选择合适的人员复杂保存和检测等），传输安全（防止数据窃取和数据篡改）。

敏捷

开发流程工具化，运维部署的自动化，监控报警的及时化，解决问题的快速化

如果加入了HTTPS怎么处理

在第三步之后

3.1 建立SSL加密通道

c->密钥算法套件（加密协议及版本）->s
c<-返回确认符合的加密协议、版本和证书(pub_key)<-s
c->使用根证书验证证书合法性，生成随机串，利用服务端pub_key RSA加密，生成握手信息，加密握手信息->s
s<-解密握手信息，获得随机串，相同的加密方式发给客户端<-s
c->解密验证，完成握手。进行通信->s

使用RSA非对称算法，服务端向客户端发送公钥，公钥包含了域名、颁发机构、过期日期。保证了公钥的合法性和服务端的身份正确性（而不会被黑客冒充）客户端向第三方验证公钥的合法性，验证通过后向服务端约定对称加密的随机字符号。保证了随机字符串只有通信双方知道。接下来的通信就使用这个随机字符号进行加密通信。因为随机字符串只有双方知道，所以信息不会被截获。
